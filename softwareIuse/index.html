Uses

Everything here earns its place. If a tool doesn't stay small, predictable, and honest about what it does — it's out.

Hardware

Latitude E5470

My main device, running Arch Linux as the only operating system. Studying, coding, all my academic stuff, and of course personal use — everything happens here.

You may ask why Arch specifically? When I choose an OS to live in, it has to satisfy all of these:

  • Kernel + core utils. NO PROGS. (Base image)
  • Open source and free
  • No corp.
  • Does not change UI/look/workflow on its own
  • Does not force updates and upgrades
  • Does not control me
  • Customizable
  • Minimal rolling release
  • Easy to use (does not mean easy to learn*)

Each point could take pages, so headlines only. Arch satisfies all of them — and I've had no problems with systemd (till now), so no need to jump to void.

I use dwl, foot, and wmenu, deliberately following in the footsteps of dwm, st, and dmenu, out of a deep appreciation for the suckless philosophy. I value software that stays small, predictable, and honest about what it does. Minimal storage, minimal CPU and memory usage, and minimal distractions. A workflow that does exactly what I tell it to, and nothing more.

RIP X11, you served your time.

Software

Wayland-based minimalist stack

In pursuit of a lightweight, efficient, distraction-free environment, I deliberately avoided full desktop environments such as GNOME, KDE Plasma, or even lightweight stacks like Xfce. Instead, I built my system around the Wayland display protocol — a modern, secure, low-latency alternative to the traditional X server model.

At the core is dwl, a dynamic tiling Wayland compositor inspired by dwm. dwl preserves the minimal philosophy of suckless tools while leveraging Wayland's cleaner architecture. My configuration focuses strictly on functionality, with only essential patches applied to support my daily workflow.

wmenu

A minimal Wayland-native menu inspired by dmenu. It integrates seamlessly with dwl and avoids any XWayland dependency, keeping the whole graphical stack purely Wayland. I use it to trigger scripts and present minimal selection menus — piping shell output straight into it makes it an essential component in several automation workflows.

foot

My terminal: fast, lightweight, Wayland-native. Given that most of my interaction with the system happens through the terminal, foot is the central interface of my usage.

vim

The editor I spend the majority of my time in — a complete working environment for programming, note-taking, configuration, and documentation. Modal editing lets me operate with minimal context switching. Config lives at .vim/vimrc in my dotfiles.

swhkd

Global key bindings under Wayland. A hotkey daemon inspired by sxhkd, designed to work cleanly with compositors like dwl. Bindings live at .config/swhkd/swhkdrc.

mako

Notifications — lightweight, built for Wayland compositors, unobtrusive, and no heavy dependencies pulled in.

vim - LaTeX - zathura

These three combined are my studying and note-taking pipeline — summarizing lectures and managing academic work. The plugin code is at .vim/ftplugin/tex.vim in my dotfiles. I wrote all of it myself: there are better options online, but mine do exactly what I need and nothing more, which is the whole point.

GnuPG (GNU Privacy Guard)

My go-to for locking down everything — emails, photos, backups — because privacy is just common sense. No shady middlemen, no bloated software, just pure control. I trust math over people, and encryption keeps my data mine.

LibreOffice (docx, pptx, etc..)

For opening documents other people send me. For producing my own, it's vim + LaTeX + zathura, as above.

Software I was using

Kali Linux

For penetration testing, security research, and offensive security workflows I ran Kali in its default configuration, straight from the official installer. Used it for educational purposes, then returned to Arch.

The main draw is the preinstalled toolset, organized by category and integrated into the system menus: network scanning and enumeration, web application testing frameworks, exploitation and post-exploitation utilities, wireless attack tools, reverse engineering and binary analysis. A curated environment that lets me focus on methodology and analysis instead of installation and dependency management.

Suckless tools (X11 era)

Before Wayland, the same philosophy on the X server: dwm, st, and dmenu. Intentionally minimal configuration, functionality over aesthetics, only the patches my workflow actually required.

sxhkd + xclip

sxhkd handled my key bindings on X — an X daemon that executes commands on input events, which made scripts one keypress away. Combined with dmenu and xclip (CLI for X selections) it was a genuinely powerful workflow. Bindings were at .config/sxhkd/sxhkdrc in my dotfiles.
